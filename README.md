# Mova

Mova 是一个纯客户端的 Flutter 创作应用，面向“图像生成、视频生成、素材管理、任务追踪”这一整条工作流。

目前项目已经打通了：

- 基于 AgentEarth.ai 的工具发现与执行
- 图片生成工作流
- Seedance2 生视频工作流
- 素材上传、管理、分类、引用与回收
- 本地任务记录、结果追踪、失败重试与结果回看

这个项目的定位不是一个重后端平台，而是一个把密钥、工具参数、素材、任务状态都尽量收在本地侧的创作客户端。

## 项目亮点

- 纯客户端架构
  所有配置都在本地维护，适合个人工作流、内部小范围使用或快速原型验证，不依赖自建中台即可开始使用。

- 一套界面串起图片和视频创作
  当前应用同时覆盖图片生成与视频生成，两类任务共用素材库、任务队列、结果回看与导入逻辑。

- 面向 Seedance2 的多模式视频工作流
  已支持文本生视频、首帧图生视频、首尾帧图生视频、参考素材生成等模式。

- 面向图片生成的独立任务流
  图片任务与视频任务拆分管理，支持结果列表、状态追踪、失败重试、入库回流等能力。

- 素材库不是“附件列表”，而是可复用工作区
  支持图片、视频、音频素材管理，支持分类、重命名、筛选、预览、复制引用链接、删除、保存、回流创作等操作。

- 视频截帧直接接进创作流程
  可以从本地视频、云端素材或历史任务结果里截帧，并把截出的画面重新作为素材或创作输入继续使用。

- 提交前可预览工具和参数
  在真正发起任务前，可以先查看要提交给上游工具的参数结构，减少误触发和误扣费。

- 本地保存最近状态
  设置、素材、任务和最近使用记录会保存在本地，方便持续迭代使用。

- 文本生视频

### 3. 对象存储

当前项目已对接两种素材存储方案：

选择它们的原因主要是：

- 接起来相对直接

- 方便当前这类客户端工作流快速落地

但这仍然不是推荐结论，只是“目前用起来比较方便”的现实选择。购买、开通或长期使用前，请务必自行甄别：

## 安全说明

这个项目是纯客户端应用，当前 API Key、存储密钥和相关配置主要保存在本地。

这意味着它更适合：

- 个人自用
- 内部小规模试验
- 原型验证
- 对部署成本敏感、先跑通工作流的场景

但也意味着你必须注意以下风险：

- 不要把生产级高权限密钥直接发给不可信用户使用
- 不要把 Access Key / Secret Key / API Key 明文传播到公共群、公开仓库或截图里
- 如果是多人分发安装包，请默认视为“客户端配置有被提取风险”
- 尽量使用权限更小、可轮换、可撤销的密钥
- 对存储桶、回源域名、私有读写策略、跨域和签名过期时间做额外约束

如果后续准备面向更广范围分发，建议进一步演进为：

- 将高敏感凭据下沉到服务端
- 由服务端签发短时上传 / 下载凭证
- 对调用频率、用户身份和资源访问做更细的控制

- 首帧图生视频
- 首尾帧图生视频

- 七牛云配置
- Bitiful S4 配置
- 测试连接
- 本地数据导出

## 适用场景

这个项目比较适合：

- 想快速搭一个本地可运行的 AI 创作客户端
- 需要把图片、视频、素材、任务统一管理
- 希望先验证工作流，而不是先建设完整后端
- 希望在自己的设备上直接配置和测试多家上游能力

如果你的目标是：

- 面向大量终端用户发布
- 做严格的多用户权限隔离
- 在服务端统一托管密钥与审计
- 做正式生产级结算与风控

那就建议在这个客户端基础上继续补后端能力，而不是长期停留在“纯客户端持有密钥”的模式。

## 开发环境

项目基于 Flutter。

已知当前版本：

- App version: `1.0.0+1`
- Android application id: `com.jbrains.mova`
- iOS bundle id: `com.jbrains.mova`

## 本地运行

如果你的环境里没有把 Flutter 放进 `PATH`，请先使用 Flutter 可执行文件的绝对路径。

```bash
flutter pub get
flutter run
```

## 构建

Android release APK：

```bash
flutter build apk --release --split-per-abi
```

只打 `ARM64 v8a`：

```bash
flutter build apk --release --split-per-abi --target-platform android-arm64
```

iOS 无签名构建：

```bash
flutter build ios --no-codesign
```
